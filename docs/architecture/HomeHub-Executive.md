# HomeHub — 总统方案设计 (Executive Solution Design)

> 版本：v1.3 | 日期：2026-06-28
> 
> 本文档面向合伙人、投资者、技术负责人，从商业、产品、技术、实施四个维度给出项目的完整方案。

## 一、执行摘要

### 项目定位

**HomeHub** 是一个现代化、中文化、AI 驱动的家庭库存管理系统。融合 Grocy（库存管理）和 HomeBox（物品追踪）的核心能力，以全 TypeScript 技术栈重写，原生支持 NFC/RFID 物联交互和 MCP 协议 AI 集成。

### 一句话

> 家庭版 ERP + AI 管家 + 一碰/一扫即查（NFC / QR / 条码统一触发）

### 核心数据

| 指标 | 数据 |
|------|------|
| 目标市场 | 中国 4.9 亿家庭（中等收入以上约 1.4 亿户，国家统计局宽口径） |
| MVP 开发周期 | 8 周（不包含插件系统，插件化在 Phase 2） |
| 初始团队 | 2-3 人（全栈 + 前端 + 兼职硬件） |
| 技术栈成熟度 | 全部成熟（Vue/NestJS/Drizzle/Naive UI 均为活跃开源项目） |
| MVP 硬件成本 | 0（PWA + 浏览器扫描，用户无需额外硬件） |
| 触发方式 | 统一 Scanner 层：NFC（Android Web NFC + iOS NDEF URL 跳转） / QR / 条码 / RFID 同一架构 |
| 物品模型 | 通用物品 + 状态机配置，一张表管所有（食品/电池/设备/煤气罐） |
| 编码输出 | 双编码策略：HH:01 离线编码 + NDEF URL 在线回退（iOS 兼容） + HAL 硬件抽象 |
| 架构策略 | Core + Plugin Registry 设计，多语言插件（JS同进程 + Python/Go/Rust IPC via npx/uvx），Phase 2 起可插拔 |

## 二、商业论证

### 2.1 市场背景

家庭库存管理在中国是一个**被忽视但真实存在的需求**：

| 人群 | 痛点 | 规模 |
|------|------|------|
| 双职工家庭 | 不知道冰箱有什么，重复购买/浪费 | ~2 亿家庭 |
| 有老人/慢性病家庭 | 药品过期无人知，急救物品找不到 | ~1 亿家庭 |
| 母婴家庭 | 奶粉/尿布库存管理，保质期追踪 | ~5000 万家庭 |
| 智能家居爱好者 | 追求自动化，NFC/IoT 控 | ~1000 万家庭 |
| 极简主义/收纳控 | 物品定位，断舍离决策辅助 | ~2000 万家庭 |

### 2.2 为什么是现在

- **NFC 普及**：2024 年中国智能手机 NFC 渗透率约 70%，中高端机型（2000 元以上）搭载率超 85%，预计 2026 年全市场突破 85%
- **AI Agent 爆发**：MCP 协议 2025 年成为 AI 工具调用标准（已捐 Linux 基金会，9000+ 公共 Server），家庭管理是天然场景
- **PWA 成熟**：Chrome 89+ 的 Web NFC API + 离线缓存，前端能力已达原生 90%（国内浏览器碎片化需适配，见 PRD 4.11 节）
- **国产替代窗口**：Grocy/HomeBox 均为国外项目，国内虽有"收起来"等商业产品但无 NFC/AI 集成、无开源方案

### 2.3 竞品对比

| 产品 | 技术栈 | 移动端 | NFC | AI | 国内适配 | 开源 | 综合评分 |
|------|--------|--------|-----|-----|---------|------|---------|
| **Grocy** (v4.6.0) | PHP+Blade+JS (9.2k⭐) | ✅ Android/iOS 原生 | ❌ | ❌ | ⚠️ 社区中文翻译不完整 | ✅ MIT | ⭐⭐ |
| **HomeBox** (v0.26.2) | Go+Vue (6.5k⭐) | ⚠️ Web 响应式 | ❌ | ❌ | ⚠️ 有中文但无国内条码/单位 | ✅ AGPL-3.0 | ⭐⭐ |
| **收起来** | 闭源商业 | ✅ iOS+Android | ❌ | ⚠️ AI 识图 | ✅ 电商导入/国内条码 | ❌ | ⭐⭐⭐ |
| **食光冰箱/冰箱控** | 闭源商业 | ✅ iOS | ❌ | ⚠️ AI 识图 | ⚠️ 仅食品/食谱 | ❌ | ⭐⭐ |
| **HomeHub** | Vue+NestJS+Naive UI | ✅ (PWA) | ✅ | ✅ (MCP) | ✅ | ✅ 计划中 | ⭐⭐⭐⭐⭐ |

### 2.4 商业模式（长期）

| 阶段 | 模式 | 说明 |
|------|------|------|
| MVP | 免费开源 | 吸引用户，建立社区 |
| 增长 | 托管版 (SaaS) | ¥9.9/月 或 ¥99/年，含云备份 + 多端同步 |
| 扩展 | NFC 标签套件 | ¥29/套（10张标签 + 引导卡 + 贴纸） |
| 高阶 | AI Agent 增值 | 超额 AI 调用按量计费（¥0.01/次） |
| 企业 | 企业版 | 餐厅/小仓库管理场景，按席位收费 |

## 四、技术方案

### 4.1 技术选型

| 层 | 选型 | 版本 | 选择原因 |
|----|------|------|---------|
| 前端框架 | Vue 3 + Vite | >= 3.5 / >= 6 | 国内生态最成熟，Composition API |
| UI 库 | Naive UI | >= 2.40 | 类型系统最严谨，AI 生成代码质量高，响应式内置 |
| 后端框架 | NestJS | >= 11 | 模块化 + DI + 官方 MCP 支持 |
| ORM | Drizzle ORM | >= 0.40 | 类型安全，同一 Schema 适配 SQLite + PostgreSQL |
| 数据库 | PostgreSQL (默认) / SQLite (单用户备选) | — | JSONB 原生 JSON 查询、全文索引、pgvector 未来扩展；Drizzle ORM 零迁移成本 |
| AI 集成 | MCP 协议 | — | 不绑定模型，对接任意 Agent |
| 物联 | Web NFC + MQTT | — | 零硬件成本入门 |
| 认证 | JWT + API Token | — | 双 Token 安全 + MCP 无状态调用 |

### 4.2 数据库策略

```
              ┌──────────────────────────┐
              │       应用代码            │
              │   (NestJS Services)       │
              └────────────┬─────────────┘
                           │
              ┌────────────▼─────────────┐
              │    Drizzle ORM Query      │
              │    抽象 SQL 生成层         │
              └──────┬──────────┬────────┘
                     │          │
           ┌─────────▼──┐  ┌───▼──────────┐
           │ PostgreSQL │  │  SQLite      │
           │  (默认)    │  │  (单用户备选) │
           │  JSONB     │  │              │
           │  全文索引   │  │              │
           │  pgvector  │  │              │
           └────────────┘  └──────────────┘
              主推部署         自托管备选
```

切换方式：改一行 `drizzle.config.ts` 的 `dialect` 字段 + 环境变量 `DB_TYPE`。Schema 定义完全不变。

### 4.3 NFC 方案

| 维度 | 方案 |
|------|------|
| 标签类型 | NTAG213/215 (¥0.3-0.8/张) |
| 读取方式 | Web NFC API (Android Chrome >= 89) |
| 标签内容 | NDEF URL: `https://homehub.app/nfc/{familyCode}/{tagUid}` |
| iOS 方案 | MVP 阶段用二维码兜底，Phase 2 加 Flutter 壳 |
| 写入工具 | Node.js NFC 写入 CLI + 批量打印工具 |

### 4.4 MCP 协议设计

```
MCP Server (内置 Plugin — builtin.mcp-server)
├─ Transport: streamable-http (SSE)
├─ Auth: Bearer <api_token>
├─ 认证来源: api_tokens 表 (SHA-256)
└─ Tools: 5 类 23 个
    ├─ 库存类: search_items, add_item, consume_item, move_item, update_item, delete_item
    ├─ 清单类: get_lists, create_list, add_to_list, check_list_item, uncheck_list_item, assign_list_item, get_my_tasks
    ├─ 食谱类: get_recipe_recommendations, search_recipes, get_meal_plan
    ├─ 统计类: get_stock_summary, get_expiring_items, get_waste_report, get_statistics
    └─ 配置类: bind_trigger, get_bindings, create_automation
```

> **设计原则：** MCP 只暴露纯净的领域操作。语音/图片等多模态输入由 AI Agent 自行处理（ASR、图像识别），处理完成后调用 CRUD 工具写入结构化数据。HomeHub 不负责理解语音和图片。

详见 [HomeHub-PRD.md](./HomeHub-PRD.md) → 4.7 MCP 接口设计。

## 七、为什么是现在

1. **技术窗口**：Web NFC API + MCP 协议 + PWA 三者在 2025-2026 同时成熟
2. **市场窗口**：Grocy 2025 年停止活跃开发，大量用户寻找替代品
3. **AI 窗口**：大模型 Agent 爆发但缺乏家庭场景的数据入口，HomeHub = MCP 即数据管道
4. **硬件窗口**：NFC 贴纸成本降至 ¥0.3/张，中国手机 NFC 渗透率约 70%（中高端 > 85%）
