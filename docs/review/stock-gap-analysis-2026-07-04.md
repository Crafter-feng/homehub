# 库存管理模块差距分析 — HomeHub vs Grocy

> 对标项目：[Grocy](https://github.com/grocy/grocy)
> 
> 分析日期：2026-07-04
> 
> 范围：Stock Overview 页面及库存管理核心流程

---

## 已实现且基本对齐的功能

- 物品 CRUD / 消耗 / 入库 / 转移位置
- 搜索过滤（名称/品牌/条码 + 类别/位置/状态）
- 动态分组（8 种维度，折叠展开，localStorage 持久化）
- 表格/卡片双视图 + 列可见性控制
- 行内快捷操作（消耗/入库/标记已开）+ 弹窗确认防误触
- 过期/临期/低库存预警着色和横幅
- 统计卡片（总数/临期/过期/低库存/库存价值）
- 价格统计（加权均价/历史最高最低）
- 操作历史时间轴 + 全局流水日志页
- 快速入库 3 步流程 + 扫码支持
- 产品主数据管理 + 条码
- CSV 导入导出（后端已就绪）
- 盘点模式（差异比对，前端展示）

## 差距清单

### P0 — 核心缺失，影响基本使用体验

| # | 差距 | Grocy 实现 | HomeHub 现状 | 说明 |
|---|------|-----------|-------------|------|
| 1 | 库存条目级管理 | 每次入库生成独立 stock entry（独立到期日/价格/开启状态），`stock_next_use` 视图管理 | 产品级累加数量，无批次概念 | 无法追踪同一产品的不同批次，是库存系统从"清单"升级为"管理系统"的关键 |
| 2 | FIFO 先进先出消耗 | 按 `stock_next_use` 视图自动从最早到期条目消耗 | 无此逻辑 | 无批次则无法实现 FIFO |
| 3 | 盘点结果写回后端 | 盘点差异直接调用 adjust API 写回 | 仅前端展示差异，不调用 adjust API | 盘点形同虚设 |
| 4 | 操作撤销 (Undo) | 支持撤销任何交易（含关联交易），每次操作后弹出 toast | 仅留了注释框架 | 误操作无法恢复 |
| 5 | 库存日志页 (Stock Journal) | 独立 `/stockjournal` 页面，按产品/类型/位置/用户/时间筛选 | `/history` 页功能较粗糙 | 无法高效追溯操作 |

## 建议实施路径

### 第一批（P0）— 从"清单"到"管理系统"

条目级管理 + FIFO 消耗 + 盘点写回 + 操作撤销。这 4 项是库存系统质变的关键：
- 条目级管理是所有高级功能（FIFO、批次、条目编辑、冷冻调整）的数据基础
- 盘点写回让盘点功能真正闭环
- 撤销操作消除误操作焦虑

### 第二批（P1）— 补齐日常高频需求

变质标记 + 到期日自动计算 + 开启/冷冻到期调整 + 多量化单位 + CSV/批次前端入口。
这些是日常使用中最常触达的功能，完成后基本覆盖 80% 的使用场景。

### 第三批（P2）— 按场景逐步引入

根据实际使用场景决定优先级。建议优先考虑：
- 产品图片（视觉体验提升大）
- 多用户追踪（多人家庭场景）
- 购物列表关联（形成消耗→补货闭环）
